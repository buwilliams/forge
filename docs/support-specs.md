# Spec Support in Forge

Forge now includes native support for spec management — creating, organizing, and running structured specifications before any implementation begins. This replaces the need to maintain a separate spec tool like spec-kit for most projects.

## Spec types

There are three kinds of specs, each serving a distinct purpose:

### Constitution (`/.forge/constitution.md`)

The **constitution** defines non-negotiable principles for the entire project. Every Forge task is generated with these rules injected as Global Constraints. Once written, the constitution governs all future work.

Good constitution entries are checkable assertions:
- "No external dependencies without explicit approval"
- "All user-facing text reviewed before shipping"
- "Secrets never stored in source control"

### Product spec (`/.forge/product.md`)

The **product spec** captures the what and why — non-technical requirements that every piece of work must serve. It defines vision, target users, success criteria, and scope boundaries. It is not a technical document; it is the lens through which technical decisions are evaluated.

The product spec informs `/forge:new` so that new project specs stay mission-aligned. If a proposed project spec conflicts with the product's vision or scope, Forge flags it during creation.

### Project specs (`/.forge/<num>_<name>/project.md`)

**Project specs** define individual goals — the big hairy things that get decomposed into tasks and executed. Each spec is a single `project.md` file: the user's design at the top, Forge's generated execution config (global constraints, dynamic verification, execution commands) appended at the bottom when the spec is set up.

Project specs live in numbered directories under `.forge/`:
```
.forge/
  00001_auth_system/
    project.md        ← the project spec (user design + Forge config)
    council.md        ← generated by Forge
    verifier.md       ← generated by Forge (project-specific verifier)
    plan.md           ← generated by Forge
    todo/             ← pending tasks
    working/          ← in-progress task
    done/             ← completed tasks
    blocked/          ← blocked tasks
    council/          ← generated role agents
```

The numeric prefix (`00001`, `00002`, …) encodes creation order, the same way task files are numbered within a run.

**Two templates are available** depending on project type:

- `project-technical.template.md` — for software, infrastructure, or tooling. Includes testing requirements, code quality constraints, performance targets, and security considerations.
- `project-general.template.md` — for non-technical deliverables like reports, strategies, playbooks, or content. Includes stakeholders, success criteria, and resource constraints.

Forge detects the project type automatically by scanning for tech stack files (`package.json`, `Cargo.toml`, etc.). If ambiguous, it asks.

## Commands

### `/forge:setup`

Initializes the project by creating `constitution.md` and optionally `product.md` through a guided conversation.

```
/forge:setup
```

Forge reads the current project context (README, package.json, CLAUDE.md, etc.) to ask informed questions rather than presenting blank forms. You answer conversationally; Forge drafts the document. When you're satisfied, type `accept` to write the file.

If either file already exists, Forge displays it and asks whether to keep or update it.

### `/forge:new-spec <work-name>`

Creates a new numbered project spec through a guided conversation.

```
/forge:new-spec auth-system
/forge:new-spec q3-marketing-report
```

Forge:
1. Assigns the next spec number (`00001`, `00002`, …)
2. Detects whether this is a technical or general project
3. Reads the constitution and product spec for context
4. Guides you through filling in the spec conversationally, flagging anything that conflicts with the constitution or product vision
5. Writes `project.md` to `.forge/<num>_<slug>/` when you type `accept`, then automatically runs the full setup pipeline (council, spec agent, role agents, verifier, task decomposition)

### `/forge:list`

Displays incomplete specs by default — those with work remaining or not yet run. Pass `--all` to include completed specs.

```
/forge:list          # show incomplete specs only
/forge:list --all    # show all specs including completed
```

Output includes:
- Whether `constitution.md` and `product.md` exist
- Each spec directory, its title, and its status:
  - `not started` — spec written, Forge has never run on it
  - `in progress` — tasks in todo/ or working/
  - `blocked` — all remaining tasks are blocked
  - `partial` — some done, some blocked
  - `done` — all tasks completed (hidden unless `--all`)

### `/forge:start <work-name>`

Runs the full Forge pipeline on a named project spec — setup if needed, then execution.

```
/forge:start auth-system
/forge:start auth-system --ask
/forge:start auth-system --clean
```

It resolves the spec directory by name or number, checks what setup has already been done, resumes from the right point, and executes all tasks through the implement → verify → commit loop.

Work-name matching is flexible:
- `auth-system` → matches `00003_auth_system`
- `00003` → matches `00003_auth_system`
- `auth` → matches `00003_auth_system` if unambiguous

Flags:
- `--ask` — interactive approval at each setup phase
- `--clean` — clear forge state and start over from setup

### `/forge:remove <work-name>`

Deletes a project spec directory after confirmation.

```
/forge:remove auth-system
```

Forge displays the spec, its current status, and how much work has been done — then requires you to type `confirm` before deleting. Cancels on any other input.

## Workflow

```
/forge:setup                       # set up constitution + product spec (once per project)

/forge:new-spec auth-system        # create a project spec (guided conversation + auto setup)
/forge:new-spec data-export        # create another

/forge:list                        # see what's there

/forge:start auth-system           # execute tasks for the first spec
/forge:start data-export --ask     # run the second with approval gates at each phase

/forge:list                        # check progress

/forge:remove old-experiment       # clean up a spec you no longer need
```

## Relationship to spec-kit

If you use [spec-kit](https://github.com/github/spec-kit) to produce a `spec.md`, you can still use it as a starting point. Copy or reference the spec-kit output as your `design.md` inside a Forge spec directory, or use `/forge:new` to produce the spec directly within Forge and skip spec-kit entirely.

The Forge spec system covers the same territory as spec-kit's `/speckit.constitute`, `/speckit.specify`, and `/speckit.plan` commands, with tighter integration into the Forge execution pipeline.

## Templates

Templates live in `templates/` in the Forge plugin directory. They serve as structured guides that Claude uses during conversational spec creation — they are not presented to the user directly.

| Template | Used by |
|---|---|
| `constitution.template.md` | `/forge:setup` (constitution step) |
| `product.template.md` | `/forge:setup` (product step) |
| `project-technical.template.md` | `/forge:new-spec` when tech stack detected |
| `project-general.template.md` | `/forge:new-spec` for non-technical projects |
| `verifier-technical.template.md` | Verifier generator for technical projects |
| `verifier-general.template.md` | Verifier generator for non-technical projects |
